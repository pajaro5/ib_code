COMMANDING OFFICER'S DEBRIEF

The correct problems are rows 1, 2, 3, 5, and 6. This is exactly what a database management system (DBMS) solves:

  • Schema enforcement — fixed column structure, every row follows the same format
  • Primary Keys — a unique ID per pilot, so "Maverick" can exist twice (different periods) without confusion
  • Normalisation — separate PILOT, AIRCRAFT, and MISSION tables instead of one flat file
  • Constraints — NOT NULL rules catch missing mandatory fields
  • Single source of truth — update Maverick's rank once, everywhere reflects it

MISSION 04 — THREAT ASSESSMENT ✦ PENSAMIENTO CRÍTICO
RELATIONAL MODEL.
STRENGTHS vs THREATS.

The relational model has been the backbone of data systems since 1970. But no technology is invincible. Your job as a Data Officer is to know both its STRENGTHS (why it dominates) and its THREATS (where it fails). Study the mission, then evaluate each intel item.

THREAT ASSESSMENT — CLASSIFY EACH STATEMENT
Is this a STRENGTH (benefit) or a THREAT (limitation) of the relational model?
Constraints like PRIMARY KEY, FOREIGN KEY, and NOT NULL automatically reject invalid data before it is stored.
Adding a new column to a table with 50 million rows can lock the entire table for hours in production.
ACID transactions guarantee that if a system crashes mid-operation, the database is not left in a corrupt half-updated state.
Scaling a relational database across 500 servers (horizontal scaling) is extremely complex compared to NoSQL alternatives.
Normalisation eliminates duplicate data — updating a pilot's rank requires changing exactly one row, not hundreds.
Storing unstructured data like cockpit video recordings, satellite imagery, or sensor streams is awkward in rigid fixed-column tables.

MISSION 05 — CALLSIGN BRIEFING ✦ APRENDIZAJE CON SIGNIFICADO
CALLSIGN BRIEFING.
KNOW YOUR TERMINOLOGY.

Every pilot needs to know the callsigns. In databases, the key terms are your vocabulary for the rest of the course. We'll learn them through the TOPGUN context — then test your recall immediately.

📦 ENTITY

A type of asset we track data about.
TOPGUN: PILOT, AIRCRAFT, MISSION, BASE
DB: Each entity becomes a table.

🔧 ATTRIBUTE

The specs of that asset — its properties.
TOPGUN: An aircraft's model, top speed, status
DB: Each attribute becomes a column.

👤 TUPLE

One specific unit — a single complete record.
TOPGUN: Pete "Maverick" Mitchell's full record
DB: Each tuple is one row in a table.

🔑 PRIMARY KEY (PK)

The tail number — unique, never repeated, never null.
TOPGUN: Tail# 202 identifies one specific F-14
DB: PK uniquely identifies each tuple.

🗝 FOREIGN KEY (FK)

The home base ID — links to another table.
TOPGUN: base_id in PILOT references BASE table
DB: FK enforces referential integrity.

📋 RELATION

The full roster — the complete table.
TOPGUN: All pilot records = the PILOT relation
DB: A relation is a table in the relational model.

ENTITY-RELATIONSHIP DIAGRAMS.

An ERD is your database blueprint — a visual map showing entities, their attributes, and how they relate. Before designing TOPGUN's database, you need to read the map. Study the notation, then answer the debrief questions.

▭ RECTANGLE

Represents an Entity
e.g. PILOT, MISSION, AIRCRAFT

◯ ELLIPSE

Represents an Attribute
Underlined = Primary Key attribute

◇ DIAMOND

Represents a Relationship
e.g. FLIES, ASSIGNED TO

1 : 1

One-to-One
Each pilot has exactly one ID card.

1 : M

One-to-Many
One base has many pilots.

M : M

Many-to-Many
Pilots ↔ Missions. Needs junction table.

| Mandatory

Entity MUST participate.
A loan MUST have a borrower.

○ Optional

Entity MAY participate.
A pilot may or may not be on a mission.

Crow's Foot

Lines show cardinality: one line (1) or fork (many) at each end of the relationship.
